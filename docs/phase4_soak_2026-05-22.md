# Phase 4 OSS Integration -- soak verification (2026-05-22)

Quick smoke after the 4 commits landed on `phase4-oss-integration`.
Goal: confirm both new daemons start cleanly, run their first tick
without crashing, and that `sentry_triage` absorbs the resulting
events into the queue.

## What ran

Manual launch of the two new daemons in the background (the 4 existing
daemons were already running from a prior session); no full `nemo.bat`
restart since this is an in-conversation verification.

```powershell
Start-Process python -m daemons.rss_aggregator       # PID 20012
Start-Process python -m daemons.gdelt_poller         # PID 13284
```

Both processes stayed alive for ~5 minutes of observation, completed
their first tick, then were stopped cleanly with no traceback in the
err logs.

## Findings

### RSS aggregator -- partial pass
- Typed-feed path (SEC 8-K, BEA, Fed press, FDA, Treasury, Federal
  Register) inserted 80 events in the earlier live-verify run (see
  commit `2015f40`).
- Classify-feed path (Reuters, SeekingAlpha, MarketWatch) hit the Groq
  free-tier daily token limit (TPD 100k exhausted from earlier mock-
  classifier smoke tests).
- **Graceful degradation worked**: the base `GroqModel` retry chain
  fell back from `llama-3.3-70b-versatile` to `qwen/qwen3-32b`. Two
  MarketWatch entries succeeded via the fallback model.
- No tracebacks. The daemon kept running with classify entries
  silently dropped (`dropped_classifier_none` counter increments).

### GDELT poller -- transient API rate-limit
- First tick attempted 7 watchlist tickers. GDELT 2.0 returned a
  mix of `HTTP 429` and HTML (non-JSON) responses, indicating their
  API throttled us after the earlier 2-ticker smoke (NVDA + AAPL,
  succeeded with 58 articles in 144s).
- `fetch_gdelt_articles()` handled all failures via its existing
  except block: returned `[]`, logged the error to stderr, no crash.
- 0 articles fetched this tick. Tick completed in 38s.
- No tracebacks. The flood cap was not engaged.

### Sentry triage absorption -- pass
- 13 candidates queued in `sentry_queue` within the soak window
- 1 candidate dropped (below threshold)
- 0 tracebacks in any daemon log
- The new event sources slot into the existing pipeline without
  changes to `sentry_triage` -- confirmed.

## Issues / follow-ups

1. **Groq daily token budget shared across daemons**: rss_aggregator,
   gdelt_poller, and news_watcher all classify against the same Groq
   key. If daily budget exhausts, classify-feeds degrade gracefully
   but yield 0 events for the rest of the day. Worth observing across
   3 days before tuning -- if budget chronically exhausts, raise the
   per-daemon sleep interval to 5 s (12 RPM) or upgrade Groq tier.

2. **GDELT 2.0 API rate-limits**: hitting 7 tickers concurrently is
   too aggressive for their public endpoint. The daemon retries once
   per call on 429 and gives up; consider lowering CONCURRENT_QUERIES
   from 4 to 2 or 1 in production, or staggering ticker queries.

3. **Earlier mock-classifier smoke test events** (gdelt:NVDA, gdelt:
   AAPL, 56 rows) are still in events_store. They will eventually age
   out of the unprocessed window. Not blocking, just noise during
   the first 24h of production runtime.
